03 Auditoria de Segurança Detalhada
3.1 Autenticação e Controle de Acesso CRÍTICO

O CredentialsProvider configurado como "Test Account" permite que qualquer pessoa acesse qualquer conta sabendo apenas o e-mail. Não há senha, não há verificação de propriedade.

  • Bypass total de autenticação: POST /api/auth/callback/credentials aceita qualquer email.
  • Ausência de verificação de role nas Server Actions.
  • IDOR potencial: IDs de recursos como battleId não são verificados como pertencentes ao usuário da sessão.
3.2 Integridade do Jogo — Anti-Cheat CRÍTICO

A arquitetura atual delega ao frontend a responsabilidade de informar se o jogador ganhou ou perdeu. O backend registra o resultado sem validar.

❌ Fluxo Atual (Inseguro)

  1. Frontend inicia batalha
  2. Frontend recebe perguntas + respostas (gabarito incluso)
  3. Frontend "verifica" se acertou
  4. Frontend envia resultado ao backend
  5. Backend credita XP/Coins sem questionar

⚠ Qualquer usuário técnico pode pular etapas 1–4.

✅ Fluxo Correto (Seguro)

  1. Frontend solicita início de batalha
  2. Server gera questões, salva gabarito no DB
  3. Server retorna apenas opções (sem gabarito)
  4. Frontend envia resposta escolhida pelo usuário
  5. Server verifica no DB, registra acerto/erro
  6. Server calcula recompensa e credita

✓ Gabarito nunca chega ao cliente.

3.3 Rate Limiting e Proteção contra Abuso CRÍTICO

Sem rate limiting, o sistema está exposto a farm automatizado e ataques de custo.

4.2 Aspectos Jurídicos do Web3 / Tokens CRÍTICO

A promessa de "ativos com valor real" sem base jurídica clara é o maior risco legal do produto.

  • Contratar consultoria jurídica especializada antes do lançamento público.
  • Reclassificar tokens como "pontos de fidelidade sem valor de mercado garantido".
  • Adicionar disclaimers em todos os pontos que mencionam "valor real".
05 Análise de Arquitetura e Qualidade de Código

✅ Pontos Fortes

  • Next.js 15 App Router + Server Actions reduz surface de ataque.
  • Prisma ORM previne SQL injection por padrão.
  • PostgreSQL é a escolha certa para o modelo de dados.
  • Custodial Wallet no servidor é correto para o público-alvo.
  • Serverless na Vercel: escalabilidade automática e zero custo inicial.

🚫 Débito Técnico Crítico

  • Zod presente mas não aplicado em todas as Server Actions.
  • Sistema de erros sem padronização.
  • Zero observabilidade — operação às cegas.
  • Batalhas abandonadas sem limpeza automática.
  • Minting de NFT com falha silenciosa sem retry.
